SQL Interview Questions

Topics Covered in SQL Interviews for Data Engineers​

SQL interview questions for data engineers can vary from short-response explanations and definitions, to writing intermediate-to-advanced SQL queries. Here are the most frequently asked SQL topics:

  • Database design - These questions assess your understanding of data architecture and design. Typically, they ask you to design databases from scratch, and include SQL data manipulation and data definition statements.
  • ETL SQL questions - A frequently asked topic in data engineering interviews, these questions ask you to perform ETL tasks, e.g. extracting, transforming and loading data. These questions require proficiency in DDL and DML statements.
  • Definition-based SQL questions - Short basic SQL questions like “What’s the difference between WHERE and HAVING?”
  • Reporting and metrics - Writing queries to pull metrics. These questions will have a clearly defined metric to pull, and they can range from beginner to advanced.

Data Engineer Database Design SQL Questions​

Database design questions are common in SQL interviews for data engineering roles. You should be prepared to design the database schema for an application. One step further, you might be asked to write a query to produce a metric, based on the schema you proposed.

Some database design concepts to study for data engineering interviews include:

  • Entity-relationship modeling
  • Normalization and denormalization trade-offs
  • Dimensional modeling

Is it safe to use rowid to locate a record in oracle sql queries?​

ROWID is the physical location of a row. We can do very fast lookup based on ROWID. In a transaction where we first search a few rows and then update them one by one, we can use ROWID. What is the difference between delete and truncate in sql?​

Main differences between DELETE and TRUNCATE commands are: DML vs. DDL: DELETE is a Data Manipulation Language (DML) command. TRUNCATE is a Data Definition Language (DDL) command.

Number of Rows: We can use DELETE command to remove one or more rows from a table. TRUNCATE command will remove all the rows from a table. WHERE clause: DELETE command provides support for WHERE clause that can be used to filter the data that we want to delete. TRUNCATE command can only delete all the rows. There is no WHERE clause in TRUNCATE command. Commit: After DELETE command we have to issue COMMIT or ROLLBACK command to confirm our changes. After TRUNCATE command there is no need to run COMMIT. Changes done by TRUNCATE command can not be rolled back.

What is the difference between ddl and dml commands in sql?​

Main differences between Data Definition Language (DDL) and Data Manipulation Language (DML) commands are:

  • DDL vs. DML: DDL statements are used for creating and defining the Database structure. DML statements are used for managing data within Database. Sample Statements: DDL statements are CREATE, ALTER, DROP, TRUNCATE, RENAME etc. DML statements are SELECT, INSERT, DELETE, UPDATE, MERGE, CALL etc.
  • Number of Rows: DDL statements work on whole table. CREATE will a create a new table. DROP will remove the whole table. TRUNCATE will delete all records in a table. DML statements can work on one or more rows. INSERT can insert one or more rows. DELETE can remove one or more rows.
  • WHERE clause: DDL statements do not have a WHERE clause to filter the data. Most of DML statements support filtering the data by WHERE clause.
  • Commit: Changes done by a DDL statement can not be rolled back. So there is no need to issue a COMMIT or ROLLBACK command after DDL statement. We need to run COMMIT or ROLLBACK to confirm our changed after running a DML statement.
  • Transaction: Since each DDL statement is permanent, we can not run multiple DDL statements in a group like Transaction. DML statements can be run in a Transaction. Then we can COMMIT or ROLLBACK this group as a transaction. E.g. We can insert data in two tables and commit it together in a transaction.
  • Triggers: After DDL statements no triggers are fired. But after DML statements relevant triggers can be fired.

What is the difference between primary key and unique key in sql?​

Main differences between Primary key and Unique key in SQL are:

  • Number: There can be only one Primary key in a table. There can be more than one Unique key in a table.
  • Null value: In some DBMS Primary key cannot be NULL. E.g. MySQL adds NOT NULL to Primary key. A Unique key can have null values.
  • Unique Identifier: Primary Key is a unique identifier of a record in database table. Unique key can be null and we may not be able to identify a record in a unique way by a unique key
  • Changes: It is not recommended to change a Primary key. A Unique key can be changed much easily.
  • Usage: Primary Key is used to identify a row in a table. A Unique key is used to prevent duplicate non-null values in a column.

How mysql optimizes distinct?​

DISTINCT is converted to a GROUP BY on all columns, DISTINCT combined with ORDER BY will in many cases also need a temporary table. How mysql optimizes limit?​

In some cases MySQL will handle the query differently when you are using LIMIT # and not using HAVING:

  • If you are selecting only a few rows with LIMIT, MySQL will use indexes in some cases when it normally would prefer to do a full table scan.
  • If you use LIMIT # with ORDER BY, MySQL will end the sorting as soon as it has found the first # lines instead of sorting the whole table. When combining LIMIT # with DISTINCT, MySQL will stop as soon as it finds # unique rows. In some cases a GROUP BY can be resolved by reading the key in order (or do a sort on the key) and then calculate summaries until the key value changes. In this case LIMIT # will not calculate any unnecessary GROUP BY"s. As soon as MySQL has sent the first # rows to the client, it will abort the query. LIMIT 0 will always quickly return an empty set. This is useful to check the query and to get the column types of the result columns. The size of temporary tables uses the LIMIT # to calculate how much space is needed to resolve the query.

What is lock escalation?​

Lock escalation is the process of converting a lot of low level locks (like row locks, page locks) into higher level locks (like table locks). Every lock is a memory structure too many locks would mean, more memory being occupied by locks. To prevent this from happening, SQL Server escalates the many fine-grain locks to fewer coarse-grain locks. Lock escalation threshold was definable in SQL Server 6.5, but from SQL Server 7.0 onwards it"s dynamically managed by SQL Server.

DELETE TABLE is a logged operation, so the deletion of each row gets logged in the transaction log, which makes it slow. TRUNCATE TABLE also deletes all the rows in a table, but it won"t log the deletion of each row, instead it logs the deallocation of the data pages of the table, which makes it faster. 1. What is the primary key in SQL, and why is it important?​

In SQL (Structured Query Language), a primary key is a field or a combination of fields that uniquely identify each row in a table. A primary key is important because it ensures that there are no duplicate rows in a table, and it allows for efficient querying and indexing of the table.

For example, let’s say you have a table of employees, and you want to ensure that each employee has a unique identifier. You might create a primary key field called “employee_id.” This field would be an integer that increments by one for each new employee added to the table. When you create the primary key, you would also set a constraint that ensures that the “employee_id” field cannot contain null values or duplicates.

Having a primary key also allows for efficient indexing of the table, which can improve query performance. When you query the table using the primary key, the database engine can quickly locate the exact row you’re looking for, without having to scan the entire table.

Overall, a primary key is a crucial component of a well-designed database schema, as it ensures data integrity and enables efficient querying and indexing.

2. What is a foreign key in SQL, and how is it used to establish relationships between tables?​

In SQL, a foreign key is a column or a set of columns that refer to the primary key or a unique key of another table. It is used to establish a relationship between two tables by enforcing referential integrity between them.

When a foreign key is added to a table, it creates a link between the data in that table and the data in another table. This link ensures that the data in the two tables are always consistent with each other. For example, let’s say we have two tables, one for orders and another for customers. We can create a foreign key in the orders table that references the customer ID in the customers table. This establishes a relationship between the two tables based on the customer ID.

To illustrate this further, consider the following example:

We have two tables: one for employees and another for departments. The employee table has a foreign key column that references the department ID column in the departments table.

CREATE TABLE departments( 
    dept_id INT PRIMARY KEY, 
    dept_name VARCHAR(50) NOT NULL 
);

CREATE TABLE employees (
     emp_id INT PRIMARY KEY,
     emp_name VARCHAR(50) NOT NULL,
     dept_id INT, 
    FOREIGN KEY (dept_id) REFERENCES departments(dept_id));

In this example, the departments table contains information about the different departments in a company, while the employees table contains information about the employees in the company. The foreign key in the employees table ensures that each employee is associated with a department that exists in the departments table.

By enforcing referential integrity between the two tables, the foreign key prevents data inconsistencies and ensures data accuracy. For example, if we try to delete a department from the departments table that has employees associated with it in the employees table, the database will prevent us from doing so, as it violates the referential integrity established by the foreign key.

In summary, a foreign key is a column or set of columns in a table that establishes a relationship with another table based on the primary key or unique key of that table. It ensures data consistency and accuracy by enforcing referential integrity between the two tables.

4. What is the GROUP BY clause in SQL, and how is it used?​

The GROUP BY clause is a SQL statement used to group rows with the same values in one or more columns into summary rows, such as “find the total sales for each product”.

The GROUP BY clause is used with the SELECT statement and requires at least one aggregate function, such as SUM, COUNT, AVG, MAX, or MIN, to be used in the SELECT statement. The GROUP BY clause is usually followed by the name(s) of the column(s) that you want to group the data by.

For example, if you have a table named “Sales” with columns “Product”, “Date”, and “Sales Amount”, and you want to find the total sales for each product, you can use the following SQL query:

SELECT Product, SUM([Sales Amount]) as TotalSales
FROM Sales
GROUP BY Product;

This will return a table with two columns: “Product” and “TotalSales”, where each row represents a unique product and its total sales.

The GROUP BY clause can also be used with multiple columns, which will group the data by each unique combination of the columns. For example:

SELECT Product, Date, SUM([Sales Amount]) as TotalSales
FROM Sales
GROUP BY Product, Date;

This will return a table with three columns: “Product”, “Date”, and “TotalSales”, where each row represents a unique combination of product and date, and its total sales.

In summary, the GROUP BY clause is used in SQL to group rows with the same values in one or more columns into summary rows, using aggregate functions to perform calculations on the grouped data.

5. What is a self-join in SQL, and when would you use one?​

A self-join in SQL is a type of join operation where a table is joined with itself. This is useful when you have a table with related data in different rows that you want to join together based on a common field.

For example, consider a table called “employees” with columns for employee ID, name, and manager ID. The manager ID column contains the ID of the employee’s manager. To get a list of all employees with their manager’s name, you can use a self-join.

SELECT e.name AS employee_name, m.name AS manager_name

In this query, we’re joining the “employees” table with itself using the “manager_id” column to match each employee with their manager. The resulting table will have columns for the employee’s name and their manager’s name.

Self-joins can also be used to find relationships between data within the same table. For example, if you have a table of products with columns for product ID, name, and parent product ID, a self-join can be used to find all child products of a given parent product.

6. What is the difference between an inner join and an outer join in SQL?​

In SQL, joins are used to combine data from multiple tables based on a common column or relationship. The two most common types of joins are inner joins and outer joins.

An inner join returns only the matching rows from both tables based on the join condition. For example, if we have two tables A and B, and we perform an inner join using a common column C, only the rows where C matches in both tables will be returned.

Here is an example query for an inner join:

SELECT A.column1, B.column2
FROM A
INNER JOIN B
ON A.C = B.C;

On the other hand, an outer join returns all rows from one table and the matching rows from the other table. If there are no matching rows in the second table, the result will contain NULL values for all columns from that table. Outer joins are further classified into left outer join, right outer join, and full outer join.

Here is an example query for a left outer join:

SELECT A.column1, B.column2
FROM A
LEFT OUTER JOIN B
ON A.C = B.C;

This query will return all rows from table A and matching rows from table B based on column C. If there are no matching rows in table B, the result will contain NULL values for column2.

In summary, the main difference between inner join and outer join is that inner join only returns matching rows, while outer join returns all rows from one table and matching rows from the other table.

7. What is the difference between a correlated and an uncorrelated subquery in SQL?​

In SQL, a subquery is a query that is nested within another query, and it can be either correlated or uncorrelated. The main difference between the two is how they reference the outer query.

An uncorrelated subquery is a subquery that can be evaluated independently of the outer query. It does not reference any column from the outer query and can be executed first before the outer query. For example, consider the following query that uses an uncorrelated subquery to retrieve the average salary of employees:

SELECT first_name, last_name, salary
FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);

The subquery (SELECT AVG(salary) FROM employees) is an uncorrelated subquery because it can be executed independently of the outer query. It retrieves the average salary of all employees, and the result is used in the WHERE clause of the outer query to filter out the employees whose salary is greater than the average.

A correlated subquery, on the other hand, is a subquery that references one or more columns from the outer query. The subquery is evaluated for each row of the outer query, using the values from the outer query in the subquery. For example, consider the following query that uses a correlated subquery to retrieve the total sales for each employee:

SELECT first_name, last_name, 
  (SELECT SUM(amount) FROM sales WHERE sales.employee_id = employees.employee_id) AS total_sales
FROM employees;

The subquery (SELECT SUM(amount) FROM sales WHERE sales.employee_id = employees.employee_id) is a correlated subquery because it references the employee_id column from the outer query. It is evaluated for each employee in the outer query, and the result is used to calculate the total sales for each employee.

In summary, the main difference between correlated and uncorrelated subqueries in SQL is how they reference the outer query. An uncorrelated subquery can be evaluated independently of the outer query, while a correlated subquery is evaluated for each row of the outer query using values from the outer query.

8. What is a common table expression (CTE) in SQL, and how is it used?​

A common table expression (CTE) is a temporary named result set that can be used within a single SQL query. It allows the user to define a subquery that can be referenced multiple times within a larger query.

CTEs are defined using the “WITH” keyword, followed by the name of the CTE and the subquery that defines it. The syntax for a CTE is as follows:

WITH cte_name AS (
    SELECT column1, column2, ...
    FROM table_name
    WHERE condition
)
SELECT *
FROM cte_name

Once defined, the CTE can be used in subsequent queries as if it were a table. This can be useful in situations where a subquery needs to be referenced multiple times in a larger query, as it can simplify the syntax and improve readability.

CTEs can also be recursive, allowing for hierarchical data to be queried in a more efficient manner. A recursive CTE includes an anchor member and a recursive member, and can be used to traverse a hierarchy until a certain condition is met.

Overall, CTEs are a powerful feature of SQL that can help simplify complex queries and improve performance.

9. What is the difference between a DELETE and a TRUNCATE statement in SQL?​

In SQL, both the DELETE and TRUNCATE statements are used to remove data from a table, but they differ in their functionality and impact on the table.

The DELETE statement is used to delete specific rows from a table based on a condition specified in the WHERE clause. It can also be used to delete all rows from a table without any condition specified. The DELETE statement removes rows one by one, which can be a slow process for large tables.

On the other hand, the TRUNCATE statement is used to remove all rows from a table in one go. It is a faster method of removing data as compared to DELETE, as it deallocates the data pages and releases the disk space used by the table. However, TRUNCATE does not allow the use of a WHERE clause and cannot selectively remove specific rows.

Another difference between DELETE and TRUNCATE is that DELETE can be rolled back using a transaction log, while TRUNCATE cannot be rolled back. Once a TRUNCATE statement is executed, the data is permanently removed from the table.

In summary, if you want to selectively remove specific rows from a table or roll back the changes, you should use the DELETE statement. If you want to remove all rows from a table and release the disk space used by the table, you should use the TRUNCATE statement.

11. What is the difference between the HAVING and WHERE clauses in SQL?​

In SQL, both the HAVING and WHERE clauses are used to filter data from a query. However, there are some differences between them.

The WHERE clause is used to filter data before it is grouped or aggregated, and it is used with the SELECT, UPDATE, and DELETE statements. It filters data based on conditions that are applied to individual rows. For example, if you want to retrieve all employees who have a salary greater than $50,000, you can use the WHERE clause in a SELECT statement:

SELECT *
FROM employees
WHERE salary > 50000;

On the other hand, the HAVING clause is used to filter data after it has been grouped or aggregated, and it is used only with the SELECT statement. It filters data based on conditions that are applied to groups of rows. For example, if you want to retrieve the average salary of employees in each department and only show departments where the average salary is greater than $50,000, you can use the HAVING clause in a SELECT statement:

SELECT department, AVG(salary)
FROM employees
GROUP BY department
HAVING AVG(salary) > 50000;

In this example, the GROUP BY clause groups the data by department, and the AVG function calculates the average salary for each department. The HAVING clause filters the results to only show departments where the average salary is greater than $50,000.

In summary, the WHERE clause is used to filter individual rows before grouping or aggregating, while the HAVING clause is used to filter groups of rows after grouping or aggregating.

12. What is a window function in SQL, and how is it used?​

A window function is a type of function in SQL that performs calculations across a set of rows within a specific “window” or range. It is used to solve complex analytical problems that cannot be easily solved using simple aggregation functions.

Window functions can be used to perform calculations such as moving averages, running totals, rank, row numbering, and percentage of total. They operate on a subset of rows defined by the OVER() clause, which specifies the window or range for the function.

For example, consider a table of sales data that includes columns for date, region, product, and sales amount. A window function can be used to calculate the rolling average of sales for each product within each region over a specific time period.

Here’s an example of a window function that calculates the rolling average of sales for each product within each region over a 3-month period:

SELECT 
    date, 
    region, 
    product, 
    sales_amount,
    AVG(sales_amount) OVER (
        PARTITION BY region, product 
        ORDER BY date 
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) as rolling_avg_sales
FROM 
    sales_data

In this query, the AVG() function is used as a window function to calculate the rolling average of sales. The PARTITION BY clause divides the data into partitions or groups by region and product, and the ORDER BY clause sorts the data by date within each partition. The ROWS BETWEEN clause specifies the range of rows to include in the window frame, in this case, the current row and the two preceding rows.

Window functions can greatly simplify complex analytical queries and allow for more efficient processing of large datasets.

13. What is the difference between a transaction and a batch in SQL?​

In SQL, a transaction is a single logical unit of work that includes one or more SQL statements. A transaction is executed atomically, which means that all the statements within a transaction must either be committed or rolled back as a whole. Transactions ensure data consistency and integrity in a database by allowing multiple operations to be grouped together and executed as a single unit. If any of the statements in a transaction fails, the entire transaction is rolled back, and the database is returned to its previous state.

On the other hand, a batch is a set of SQL statements that are submitted to the database for execution as a group. Unlike transactions, batches do not offer the same level of atomicity or consistency guarantees. Each statement in a batch is executed separately, and any errors or exceptions are handled independently. Batches are typically used for tasks such as loading data into a database, running reports, or performing routine maintenance tasks.

In summary, a transaction is used to group multiple SQL statements together into a single logical unit of work that must be executed atomically, while a batch is used to submit a set of SQL statements to the database for execution as a group without the same level of transactional guarantees.

14. What is the difference between a scalar and a table-valued function in SQL?​

For example, a scalar function can be used to perform a mathematical operation, such as finding the square root of a number, or to manipulate strings, such as converting them to uppercase or lowercase.

A table-valued function, on the other hand, returns a table as its result set. This means that a table-valued function can be used in a query just like a table, allowing you to join, filter, and aggregate the data that it returns. Table-valued functions are useful for complex data manipulations where the result set is not known in advance, or for when you want to reuse a query as a table.

One key difference between scalar and table-valued functions is that scalar functions can be called inline within a query, meaning they can be used as part of a SELECT, WHERE, or ORDER BY clause. Table-valued functions, on the other hand, must be called as part of a FROM clause, since they return a table.

Another difference is that scalar functions return a single value for each row, while table-valued functions can return multiple rows. Scalar functions are generally simpler and faster than table-valued functions, but they are less flexible and cannot be used in as many situations.

In summary, scalar functions return a single value and are used to transform input values, while table-valued functions return a table and are used to manipulate and aggregate data.

Second, normalization helps to maintain data consistency and accuracy. When data is spread across multiple tables, we can establish rules and constraints that ensure data is entered correctly and meets certain standards.

Finally, normalization makes it easier to query and analyze data. By breaking down a database into smaller, more specific tables, we can retrieve and manipulate data more efficiently.

Normalization is typically accomplished through a series of steps known as normal forms. The most commonly used normal forms are the first normal form (1NF), second normal form (2NF), and third normal form (3NF). Each normal form builds upon the previous one, with each step adding more rules and constraints to ensure data integrity and consistency.
